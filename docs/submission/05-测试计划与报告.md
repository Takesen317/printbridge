## 2 测试概述

### 2.1 测试范围

| 测试类型 | 范围 | 工具 |
| --- | --- | --- |
| 单元测试 | 工具、服务、Store、Hooks | Vitest + jsdom |
| 端到端测试 | 主流程与快捷键 | Playwright(Chromium) |
| 静态检查 | 代码风格、类型 | ESLint + Prettier + tsc |
| 构建验证 | tsc + Vite + electron-builder | npm scripts |
| 手工验收 | UI、交互、跨平台 | 测试人员 |

### 2.2 不在测试范围

- DeepSeek 真实调用的服务端可用性;
- 真实印刷设备的物理输出验证(本项目为近似预览);
- 大于 8K 像素图像的极端性能(超出目标场景)。

### 2.3 测试环境

| 项 | 配置 |
| --- | --- |
| OS | Windows 11 / macOS 14+ / Linux(Ubuntu 22.04+) |
| Node | ≥ 18 LTS |
| 浏览器 | Electron 28 内置 Chromium |
| CI(可选) | 本地或 GitHub Actions |

### 2.4 通过准则

- 所有 P0 用例全部通过;
- P1 用例通过率 ≥ 95%;
- 单元测试覆盖率:`color-engine` / `print-checker` / `image-processor` / `color-convert` 行覆盖 ≥ 80%;
- 静态检查 0 error;
- 构建命令(`npm run build`)成功;
- E2E 主流程 3 个用例全部通过。

## 3 测试策略

### 3.1 测试分层

```
+------------------------+
|   E2E (Playwright)     |  → 跨模块主流程
+------------------------+
|   组件/服务层          |  → 渲染组件交互(可选)
+------------------------+
|   单元 (Vitest)        |  → 纯函数、服务、Store
+------------------------+
|   静态检查             |  → 类型 + 风格
+------------------------+
```

### 3.2 测试设计原则

- **覆盖关键路径**:颜色转换、印前检查、AI 顾问、降级路径、项目会话;
- **不依赖外部**:DeepSeek 调用使用 mock,fs/IPC 通过 jsdom 与 vi.mock 替代;
- **可重复**:所有单元测试应确定、无外部副作用;
- **可读**:用例命名遵循 `模块_场景_预期` 模式。

### 3.3 优先级

| 级别 | 含义 |
| --- | --- |
| P0 | 必须通过,阻塞发布 |
| P1 | 重要,失败需修复 |
| P2 | 一般,延期可接受 |

## 11 回归测试策略

- **触发条件**:
  - 修改主进程、Preload、Renderer 之间任一公共 API;
  - 修改 services/ 下的任一服务;
  - 修改 store 持久化结构;
  - 修改 i18n 字典结构或 menu 常量。
- **范围**:
  - 触发的模块全部用例 + E2E 主流程 + 静态检查 + 构建。
- **节奏**:
  - 每次 PR 合并前由提交人本地自测;
  - 发布前由测试负责人完整回归。
